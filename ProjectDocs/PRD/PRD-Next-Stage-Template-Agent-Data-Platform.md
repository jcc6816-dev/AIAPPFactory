## 1. 阶段性判断

当前系统已经从静态原型进入增强版 MVP 开发态，主链路具备真实演示价值：

- 可以创建表单。
- 可以保存、分享、提交。
- 可以上传文件。
- 可以执行 OCR 和 DeepSeek 结构化处理。
- 可以 Webhook 推送外部系统。
- 可以在数据库和日志中留痕。

但产品还没有进入“可市场验证的清晰版本”。主要原因不是某个单点功能缺失，而是用户从 0 开始使用时仍有学习成本，且 Agent 能力还处在轻量事件流阶段，尚未成为真正的页面工作副驾驶。

下一阶段目标应从“继续堆功能”调整为：

> 让用户从高质量模板开始，在页面级 Agent 辅助下快速改造成自己的业务场景，并完成数据收集、AI 处理和外部推送闭环。

## 2. 产品定位调整

### 2.1 不再只定位为 AI 表单生成器

“一句话生成表单”只是入口能力，不是完整产品价值。

下一阶段定位：

> AI 驱动的数据入口与自动处理平台。

用户感知路径：

```text
选择高质量场景模板
→ Agent 辅助修改
→ 发布表单
→ 收集数据 / 文件
→ OCR / LLM 自动处理
→ Webhook 推送外部系统
→ 数据查看与分析
```

### 2.2 市场验证重点

市场验证不需要证明长期平台的所有能力，而要证明三件事：

1. 用户能低成本开始使用。
2. 用户能快速得到一个看起来专业、可直接发布的业务表单。
3. 用户提交数据后能看到 AI 自动处理和外部推送的价值。

## 3. 当前已完成能力

### 3.1 已具备真实演示意义的能力

- Supabase 数据库接入。
- Development Sign In 本地登录。
- 表单生成、保存、列表、详情。
- 公开分享链接和二维码。
- 单题流填写体验。
- 文件上传和提交落库。
- Webhook 配置、推送、重试、日志。
- 飞书、钉钉、企业微信、通用 Webhook 模式。
- 百度 OCR 真实联调。
- DeepSeek 结构化处理真实联调。
- OCR 结果自动回填。
- OCR 结果进入 Webhook payload。
- Forms 首页和详情页基础指标。
- `workflow_runs` 与 `webhook_logs` 留痕。
- 页面级 Agent 模式文档。
- `/api/forms/agent` 轻量事件流雏形。

### 3.2 当前仍未完成或未稳定的能力

- 模板库。
- 模板基础上的一键创建。
- Agent 基于模板和现有 Schema 做真实增量修改。
- 数据页 Agent 的筛选、总结和异常发现。
- Google Vision OCR 真实联调。
- PDF OCR 真实处理。
- 正式 OAuth 登录和生产安全收口。
- RLS 和权限边界。
- 可部署环境检查。
- UI 统一和最终视觉定稿。

## 4. 核心调整：模板优先

### 4.1 为什么需要模板

仅靠用户输入 Prompt 从 0 生成，会带来以下问题：

- 用户不知道如何描述需求。
- 用户不知道优秀表单应该长什么样。
- 第一次生成结果不稳定，会降低信任感。
- AI 审美和字段设计不稳定，会影响付费转化。
- 销售演示时缺乏高质量样板。

模板提供高质量起点，Agent 负责修改和适配。

一句话原则：

> 模板是产品脚手架，Agent 是施工队。用户不应该从一片空地开始，而应该从一套漂亮样板房开始改。

### 4.2 模板不是字段模板，而是场景模板

模板应包含完整业务场景，而不只是表单字段。

建议结构：

```ts
type SceneTemplate = {
  id: string;
  name: string;
  industry: string;
  scenario: string;
  description: string;
  theme: FormTheme;
  formSchema: FormSchema;
  suggestedPrompts: string[];
  agentQuickActions: string[];
  ocrTemplate?: OcrTemplate;
  webhookPreset?: WebhookPreset;
  successCopy?: string;
  sampleData?: unknown;
};
```

### 4.3 模板应包含的内容

每个模板至少包含：

- 场景名称。
- 适用行业。
- 场景说明。
- 表单字段结构。
- 字段文案和 help text。
- 推荐主题。
- Agent 快捷任务。
- 是否建议 OCR。
- 是否建议 Webhook。
- 示例用途。

高价值模板可继续包含：

- 示例提交数据。
- 推荐 Webhook 平台。
- 推荐后续数据处理方式。
- 成功页文案。
- 发布页提示。

### 4.4 模板分层策略

模板库分为两条线推进，不能混在一起。

#### A. 通用增长模板

面向互联网、个人、小团队、终端用户、小型组织。

参考对象：

- v0
- Typeform
- Jotform
- Fillout
- Tally
- Paperform

目标：

- 降低首次上手成本。
- 提升首次生成质量。
- 提升模板视觉下限。
- 方便官网、社媒、活动、个人业务、小团队快速使用。
- 支撑早期市场演示和转化。

这条线由产品和研发先基于公开竞品分析推进。

#### B. 企业行业模板

面向企业、保险、金融、政企、集团内部业务。

目标：

- 贴合真实业务流程。
- 支持附件、OCR、Webhook、内部系统对接。
- 支持数据留痕和人工确认。
- 体现企业效率价值。

这条线必须基于线下调研和真实业务访谈推进，不允许研发或 AI 编程助手凭空编造行业流程。

#### C. 分工边界

通用增长模板可以由产品和研发基于公开竞品、公开使用场景和真实试用反馈先行推进，用于降低上手门槛和支撑早期演示。

企业行业模板由业务调研驱动。研发和 AI 编程助手只能负责把调研结果结构化为模板、字段、Agent 快捷任务和集成建议，不能自行假设保险、金融、政企等行业流程。

模板需求进入实现前，需要先标明来源：

- `public_benchmark`：来自公开竞品和通用互联网场景。
- `user_research`：来自真实访谈、客户反馈或线下调研。
- `internal_demo`：仅用于内部演示，不代表真实行业最佳实践。

### 4.5 第一批通用模板建议

第一批不宜过多，建议 6 到 8 个高频场景：

1. 线索收集表。
2. 联系我们表。
3. 活动报名表。
4. 活动反馈表。
5. 客户满意度调查。
6. 产品推荐问卷。
7. 预约咨询表。
8. 招聘申请表。

模板选择原则：

- 能展示表单能力。
- 能展示优秀填写体验。
- 能展示 Agent 基于模板修改。
- 能作为公开演示样板。
- 用户容易理解业务价值。

企业行业模板不进入第一批通用模板范围，待线下调研后单独立项。

## 5. 模板与 Agent 的关系

### 5.1 模板负责降低起点难度

用户可以直接选择模板创建初版表单，而不是从空白 Prompt 开始。

推荐流程：

```text
用户选择模板
→ 系统创建草稿
→ Agent 告诉用户这个模板能做什么
→ 用户用自然语言修改
→ 右侧实时预览
→ 保存 / 发布
```

### 5.2 Agent 负责个性化修改

模板提供高质量默认结构，Agent 负责根据用户需求做增量修改。

示例：

```text
用户选择“发票报销资料收集”模板。
用户：改成适合市场部活动报销。
Agent：基于模板修改字段文案，增加活动名称、预算归属、发票上传字段，并保留 OCR 模板。
```

### 5.3 模板应提供快捷任务

每个模板应内置 3 到 5 个 Agent 快捷任务。

发票报销模板示例：

- 改成适合部门报销。
- 加一个合同附件上传。
- 检查字段是否太多。
- 配置飞书机器人推送建议。
- 说明 OCR 会识别哪些内容。

会议签到模板示例：

- 改成高端沙龙报名。
- 加一个饮食偏好字段。
- 生成活动确认文案。
- 检查手机端填写体验。

## 6. 页面级 Agent 下一阶段要求

页面级 Agent 的原则以 [Page-Agent-Pattern.md](/Users/mike/Documents/AIFactory/ProjectDocs/Page-Agent-Pattern.md) 为准。

下一阶段优先实现两个页面：

1. 生成页 Agent。
2. 数据页 Agent。

### 6.1 生成页 Agent

目标：

让用户能基于模板或当前表单，用自然语言持续修改右侧表单。

必须能力：

- 从模板创建草稿。
- 基于 Prompt 生成草稿。
- 基于 existingSchema 增量修改字段。
- 校验 Schema。
- 输出变更摘要。
- 推荐字段文案优化。
- 推荐主题和布局。

验收标准：

- 用户选择模板后，右侧能立即出现可预览表单。
- 用户输入“把电话改成邮箱”，右侧字段能被修改。
- 用户输入“加一个发票上传字段”，右侧能新增字段。
- 用户输入“检查这个表单是否太长”，Agent 能输出建议但不擅自修改。

### 6.2 数据页 Agent

目标：

让用户能用自然语言理解提交数据。

必须能力：

- 总结最近提交情况。
- 根据条件筛选提交记录。
- 找出异常记录。
- 统计字段缺失情况。
- 给出导出或后续处理建议。

验收标准：

- 用户输入“找出没有上传发票的记录”，列表能筛选。
- 用户输入“总结最近 20 条提交”，Agent 能输出摘要。
- 用户输入“哪些字段经常为空”，Agent 能输出字段质量统计。

### 6.3 发布页 Agent 暂做轻量

发布页第一阶段只做：

- Webhook 配置检查。
- 常见平台配置建议。
- 失败日志解释。

真实推送测试属于高风险操作，必须用户确认。

## 7. Agent 技术要求

### 7.1 第一阶段采用 Plan-Then-Execute

暂不做复杂 Tool Calling Loop。

推荐流程：

```text
用户输入
→ 后端组织 Page Context 和可用工具说明
→ 模型输出 plan
→ 后端校验 plan
→ 后端执行工具
→ 工具结果更新页面
→ 模型或后端生成 summary
→ SSE 事件流推给前端
```

### 7.2 工具必须后端执行

模型不得直接执行工具。模型只负责：

- 理解用户意图。
- 输出计划。
- 生成总结。

工具必须由后端执行：

- 修改 Schema。
- 校验 Schema。
- 筛选数据。
- 汇总数据。
- 检查 Webhook。
- 查询日志。

### 7.3 模型和框架不得绑死业务

业务必须通过自己的协议抽象：

- Page Context 协议。
- Tool Registry 协议。
- Agent Event 协议。
- Model Provider 协议。

未来可切换：

- OpenAI。
- DeepSeek。
- Claude。
- Gemini。
- OpenRouter。
- 本地模型。
- OpenAI Agents SDK。
- LangGraph。

### 7.4 Token 成本控制

必须遵守：

- 模型只读摘要，不读全量数据。
- 数据页禁止把大量原始提交记录直接传给模型。
- 统计、筛选、校验优先由后端工具完成。
- 简单任务优先规则判断，不调用模型。
- 每次 Agent Run 限制最大步骤数、最大工具次数、最大输出长度。
- 重复摘要和字段质量检查应缓存。

推荐成本结构：

```text
规则和后端工具判断：70%
LLM 规划、解释和文案：30%
```

## 8. UI 与交互原则

### 8.1 核心 UI 由产品 / 设计侧定稿

后续核心界面视觉不依赖 AI 自由发挥。

执行方式：

- 产品或设计先确定视觉基准。
- 工程实现负责还原、组件化、响应式、状态和接口。
- AI 编程助手不得擅自大幅改动核心视觉风格。

### 8.2 模板库 UI 要降低选择成本

模板库需要让用户快速理解：

- 这个模板适合什么场景。
- 能收集什么数据。
- 是否支持 OCR。
- 是否适合 Webhook 推送。
- 可以通过 Agent 怎么修改。

### 8.3 生成页 UI 要支持三种入口

生成页应同时支持：

1. 从模板开始。
2. 从 Prompt 开始。
3. 从已有表单继续修改。

## 9. 推荐下一阶段路线

### Phase 1：模板库 MVP

目标：

让用户不再从 0 开始，而是从高质量场景模板开始。

范围：

- 定义 `SceneTemplate` 数据结构。
- 内置 6 到 8 个模板。
- 生成页支持模板选择。
- 模板可一键生成草稿。
- 模板携带 Agent 快捷任务。

不做：

- 模板市场。
- 用户自定义公开模板。
- 模板付费分层。
- 模板版本系统。

### Phase 2：模板 + 生成页 Agent 增量修改

目标：

让用户基于模板，通过自然语言持续调整表单。

范围：

- `generator.revise_form_schema`
- `generator.validate_form_schema`
- 变更摘要。
- 右侧预览更新。

不做：

- 长期记忆。
- 多 Agent 协作。
- 自动发布。

### Phase 3：数据页 Agent

目标：

让用户感受到“收集到的数据可以被 AI 理解和处理”。

范围：

- 数据摘要。
- 自然语言筛选。
- 异常识别。
- 字段缺失统计。

### Phase 4：部署与市场验证准备

目标：

让系统具备可演示、可部署、可客户试用的基础条件。

范围：

- 环境变量清单。
- Supabase RLS 策略。
- 正式登录方式。
- OCR / LLM provider 配置说明。
- Webhook 测试说明。
- 演示脚本。

## 10. 当前不做

下一阶段暂不做：

- 多租户复杂架构。
- Skill 商店。
- Workflow 可视化编排。
- 审批流。
- 模板市场。
- Agent 长期记忆。
- 多 Agent 协作。
- 自动扣费复杂策略。
- AI 自动发布。
- OpenClaw / MCP 对外开放。

这些属于长期平台能力，不进入下一阶段市场验证主线。

## 11. 成功标准

下一阶段完成后，应能支撑一次清晰的客户演示：

```text
用户选择“发票报销资料收集”模板
→ Agent 解释模板用途
→ 用户要求改成市场部活动报销
→ Agent 修改表单字段
→ 用户发布分享链接
→ 提交一张发票图片
→ 系统 OCR 识别并用 DeepSeek 结构化
→ 数据写库
→ Webhook 推送到外部系统
→ 数据页 Agent 总结提交情况
```

如果这条链路能稳定完成，说明产品已经具备市场验证价值。

## 12. 一句话结论

下一阶段不应继续从 0 到 1 地堆新功能，而应围绕：

> 模板降低起点，Agent 辅助修改，数据处理闭环证明价值。

这三件事收敛执行。
